分布式锁模块（lock）技术分享

2471字 2026-03-06
10 min read
Scroll Down

1.1 业务背景

在分布式、多实例部署下，同一份数据可能被多个节点同时修改，仅靠 JVM 内的 synchronized 或 ReentrantLock 无法跨进程互斥，会导致：

  • 重复扣减库存：多个请求同时读库存、同时扣减，造成超卖。
  • 重复发放优惠券：同一用户多请求同时通过“是否已领”判断，导致重复发放。
  • 并发更新覆盖：对同一订单/账户的并发更新互相覆盖，数据不一致。

因此需要跨 JVM、跨实例的互斥能力，即分布式锁：在同一时刻，针对同一资源（由锁 key 标识）只允许一个持有者执行临界区逻辑。

1.2 模块目标

本模块要解决的核心问题是：在分布式环境下，对“同一业务资源”提供统一的加锁/解锁能力，并支持多种锁语义（可重入、公平、读写分离），同时保证使用简单、释放可靠。

具体包括：

  • 声明式加锁：通过注解指定锁 key、锁类型、等待时间等，业务代码无侵入。
  • 多种锁类型：可重入锁、公平锁、读锁、写锁，适配不同并发场景。
  • 灵活的锁 key：支持 SpEL 从方法参数中解析 key，支持前缀与分隔符，避免 key 冲突。
  • 获取超时策略：在指定时间内未拿到锁时，可走自定义处理逻辑（默认抛异常），避免无限等待。
  • 释放可靠：AOP 在 finally 中统一解锁，避免业务异常导致锁泄漏。

2. 为什么这样设计，如何设计实现的（关键细节）

2.1 整体架构思路

  • 注解驱动：用 @ILock 声明“此方法需要分布式锁”，key、类型、超时、超时处理器均在注解上配置。
  • 锁类型与实现分离：LockType 枚举（Reentrant / Fair / Read / Write）对应四种 Redisson 锁实现，由 LockManager 统一按类型返回 ILocker，便于扩展新锁类型。
  • 统一加锁入口：AOP 拦截带 @ILock 的方法 → 解析 key → 按类型取锁 → tryLock(waitTime) → 成功则 proceed 并在 finally 中 unlock，失败则走 LockTimeoutHandler。

这样设计的原因：不同业务对“公平性”“读写分离”需求不同，用枚举 + 策略映射可以清晰扩展；key 用 SpEL 能从方法参数中抽业务主键，避免硬编码；超时后走 Handler 可自定义“抛异常 / 返回默认值 / 打日志”等，不写死在切面里。

2.2 核心流程（AOP + tryLock + finally 释放）

请求进入 → @ILock 切面
         → 从注解取 keys、prefix、separator、lockType、timeout、timeUnit、timeOutHandler
         → SpEL 解析 keys 得到 parseKeys，与 prefix 用 separator 拼成 lockName
         → lockManager.getLock(lockType) 得到 ILocker
         → lock.tryLock(lockName, timeUnit, waitTime)
         → 若成功：try { return joinPoint.proceed(); } finally { lock.unlock(lockName); }
         → 若失败：从容器取 timeOutHandler Bean，执行 handler(lockName)（默认抛 RuntimeException），不执行 proceed

要点：

  • 切面顺序：@Order(Ordered.HIGHEST_PRECEDENCE)，保证加锁优先于其他切面。
  • 只使用 tryLock(waitTime)：不传 leaseTime，锁由业务执行期间持有，在 finally 中显式 unlock，避免“执行时间不确定导致 leaseTime 设短了锁自动释放、设长了阻塞别人”的问题；未获取到锁则不走业务逻辑，由超时处理器决定是否抛异常。
  • key 拼接：prefix（可选）与 SpEL 解析出的 keys 用 separator（默认 :）拼接，例如 prefix=order、keys=#orderId → order:123，保证 key 唯一且可读。

2.3 锁类型与 AbstractLock 抽象

  • ILocker 接口：定义 getLock、lock、tryLock、unlock 等，与具体实现解耦。
  • AbstractLock：实现 ILocker，并实现 ApplicationContextAware 在启动时从容器拿到 RedissonClient；核心是 fetchLockOfType(lockKey)：根据子类返回的 supportType() 调用 Redisson 不同 API：
    • Reentrant：redissonClient.getLock(lockKey)，可重入锁。
    • Fair：redissonClient.getFairLock(lockKey)，按请求顺序获取。
    • Read：redissonClient.getReadWriteLock(lockKey).readLock()，共享读锁。
    • Write：redissonClient.getReadWriteLock(lockKey).writeLock()，独占写锁。
  • 四种具体实现：RedissonReentrantLock、RedissonFairLock、RedissonReadLock、RedissonWriteLock 仅需实现 supportType() 返回对应枚举，其余逻辑全部在 AbstractLock 中复用。

这样做的优点：新增锁类型只需新增枚举值、新增一个 AbstractLock 子类并注册 Bean，在 LockManager 和 fetchLockOfType 中增加分支即可，符合开闭原则。

2.4 LockManager 与 Bean 注册

  • LockManager：实现 ApplicationContextAware，在 setApplicationContext 中把 LockType 与对应实现类 Bean 放入 lockMap（Reentrant→RedissonReentrantLock 等），对外提供 getLock(LockType)。
  • LockAutoConfiguration：通过 spring.factories 自动装配；@ConditionalOnProperty(prefix = "framework.lock", name = "enable", havingValue = "true", matchIfMissing = true) 控制是否启用；注册 LockManager、四种锁 Bean、ILockAspect；并 @EnableConfigurationProperties(LockProperties.class) 绑定 framework.lock.enable。

2.5 超时处理与 LockTimeoutHandler

  • 注解属性：timeOutHandler 指定“获取锁超时”时的处理类，需为 Spring 容器中的 Bean，默认 LockTimeoutHandler.DefaultLockTimeoutHandler。
  • DefaultLockTimeoutHandler：handler(lockName) 中抛出 RuntimeException(String.format("%s请求频繁", lockName))，便于全局异常处理统一返回。
  • 切面逻辑：tryLock 返回 false 时，从容器取 iLock.timeOutHandler() 对应 Bean，若不存在则抛 ServiceException；若存在则调用 lockTimeOutHandler.handler(lockName)，不再执行 joinPoint.proceed()，因此默认行为是“拿不到锁就抛异常、不执行业务”。

自定义 Handler 可实现“打日志、发告警、返回默认值”等，只需实现 LockTimeoutHandler 并注册为 Bean，在注解上指定 timeOutHandler = YourHandler.class 即可。

2.6 InterruptedException 处理

  • AbstractLock.tryLock：内部调用 Redisson 的 lock.tryLock(waitTime, unit) 或 tryLock(waitTime, leaseTime, unit)，会抛出 InterruptedException；当前实现中 catch 后直接返回 false，相当于“等待被中断则视为获取失败”，由上层切面走超时处理器，避免把中断异常抛到业务层。

3. 这样设计的优点

维度 说明
业务无侵入 方法上加 @ILock 即可，key、类型、超时、超时处理均声明在注解上，加锁/解锁逻辑集中在切面。
锁类型可扩展 LockType + 多种 AbstractLock 实现 + LockManager 映射，新增锁类型只需新实现类与配置，符合开闭原则。
key 灵活 keys 支持 SpEL 数组，可从方法参数拼出业务 key；prefix、separator 便于按业务/模块划分 key 空间，避免冲突。
超时行为可配置 使用 tryLock(waitTime)，避免无限等待；超时后走 LockTimeoutHandler，默认抛异常，也可自定义为打日志、返回默认值等。
释放可靠 仅在 tryLock 成功时执行 proceed，并在 finally 中 unlock(lockName)，避免业务异常导致锁未释放。
与 Redisson 一致 底层完全基于 Redisson 的 RLock（可重入/公平/读写锁），复用其看门狗、Redlock 等能力，与项目内 cache 等模块技术栈统一。

4. 使用方法介绍

4.1 依赖与配置

  • 引入模块依赖 framework-biz-lock（该模块依赖 framework-cache，需保证 Redisson/Redis 可用）。
  • 自动装配：引入后默认启用，可通过配置关闭：
framework:
  lock:
    enable: true   # 是否启用分布式锁，默认 true

4.2 注解方式（推荐）

在需要加锁的方法上使用 @ILock，指定 keys（SpEL 数组）、lockType、timeout 等。

可重入锁示例（如下单、扣库存）：

@Service
public class OrderService {

    @ILock(keys = {"#orderId"}, prefix = "order:create", lockType = LockType.Reentrant, timeout = 10, timeUnit = TimeUnit.SECONDS)
    public void createOrder(String orderId, Order order) {
        // 同一 orderId 同一时刻只会有一个请求进入
    }
}

写锁示例（互斥写）：

@ILock(keys = {"#productId"}, prefix = "product:stock", lockType = LockType.Write, timeout = 5)
public void updateStock(String productId, int quantity) {
    // 更新库存，同一 productId 写互斥
}

读锁示例（多读单写）：

@ILock(keys = {"#productId"}, prefix = "product:info", lockType = LockType.Read, timeout = 3)
public Product getProductInfo(String productId) {
    // 多线程可并发读，与写锁互斥
}

多 key 与自定义分隔符：

@ILock(keys = {"#userId", "#couponId"}, prefix = "coupon", separator = "_", lockType = LockType.Reentrant, timeout = 5)
public void receiveCoupon(Long userId, Long couponId) {
    // lockName 示例: coupon_1001_2001
}

4.3 注解属性说明

属性 类型 默认值 说明
lockType LockType Reentrant 锁类型：Reentrant / Fair / Read / Write
prefix String "" 锁 key 前缀，可选
keys String[] 必填 业务 key，支持 SpEL，如 "#orderId"、"#request.id"
separator String ":" prefix 与 keys 之间的分隔符
timeout long 10 获取锁最大等待时间
timeUnit TimeUnit SECONDS timeout 的单位
timeOutHandler Class DefaultLockTimeoutHandler 获取锁超时时的处理类，需为 Spring Bean

4.4 自定义锁超时处理器

实现 LockTimeoutHandler 并注册为 Bean，在注解上指定 timeOutHandler：

@Component
public class CustomLockTimeoutHandler implements LockTimeoutHandler {

    @Override
    public void handler(String lockName) {
        log.warn("获取锁超时, lockName={}", lockName);
        throw new ServiceException("系统繁忙，请稍后再试");
    }
}

@ILock(keys = {"#id"}, timeOutHandler = CustomLockTimeoutHandler.class, timeout = 5)
public void doSomething(Long id) { ... }

4.5 编程式使用（可选）

若需要更细粒度控制（例如在方法中间加锁、或根据条件加锁），可直接注入 LockManager，按类型取 ILocker 后手动 tryLock/unlock：

@Service
public class ManualLockService {

    private final LockManager lockManager;

    public void doSomething(String resourceId) {
        ILocker locker = lockManager.getLock(LockType.Reentrant);
        boolean acquired = locker.tryLock("my:lock:" + resourceId, TimeUnit.SECONDS, 5);
        if (!acquired) {
            throw new ServiceException("获取锁失败");
        }
        try {
            // 业务逻辑
        } finally {
            locker.unlock("my:lock:" + resourceId);
        }
    }
}

注意：编程式使用时必须保证 finally 中 unlock，避免异常导致锁泄漏。

5. 归纳总结

  • 解决的问题：在分布式环境下，对“同一业务资源”提供跨 JVM 的互斥能力，避免重复扣减、重复发放、并发覆盖等问题；支持可重入、公平、读锁、写锁多种语义。
  • 设计要点：注解驱动 + LockType 与多种锁实现映射（LockManager）+ SpEL 解析 key + tryLock(waitTime) 与 finally 释放；超时后走可插拔的 LockTimeoutHandler，默认抛异常不执行业务。
  • 实现要点：AbstractLock 统一封装 Redisson 的 getLock/getFairLock/readLock/writeLock，子类仅声明 supportType；切面拼 lockName、按类型取锁、tryLock 成功才 proceed 并在 finally 中 unlock；InterruptedException 被捕获并转为 tryLock 失败，由超时处理器统一处理。
  • 使用方式：引入依赖后通过 @ILock(keys = {...}, lockType = ..., timeout = ...) 声明式加锁；可按需自定义 LockTimeoutHandler；需要更细粒度时通过 LockManager 获取 ILocker 手动加解锁，并务必在 finally 中释放。

本模块在保证分布式互斥、防止重复操作与并发覆盖方面可直接用于生产；使用中需注意 key 的唯一性与业务含义、timeout 与业务耗时的匹配，以及读写锁与公平锁的适用场景选择。
