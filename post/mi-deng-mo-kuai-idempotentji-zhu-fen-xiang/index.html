幂等模块（idempotent）技术分享

2324字 2026-03-06
10 min read
Scroll Down

1.1 业务背景

在并发场景下，常见两类重复执行问题：

  • REST API：用户重复点击、网络重试、前端重复提交，导致同一笔订单/支付被处理多次。
  • 消息队列：MQ 重试、重复投递导致同一条消息被消费多次，产生重复扣款、重复发货等。

若不对“同一请求/同一条消息”做幂等控制，会产生重复扣款、重复下单、重复发货等资金与数据一致性问题。

1.2 模块目标

幂等模块要解决的核心问题是：在多种业务场景下，对“同一操作”只允许被成功执行一次，后续重复请求/消息应被识别并拒绝或直接返回已有结果。

具体包括：

  • REST API 防重复提交：同一用户在短时间内的重复请求只处理一次。
  • 一次性 Token 校验：先领 Token、再带 Token 请求，保证一次一用。
  • 基于业务键的幂等：用 SpEL 从参数/消息中提取业务唯一键（如订单号、消息 ID），按键做幂等。
  • 为 MQ 场景预留扩展：通过 IdempotentScene.MQ、RepeatConsumptionException 等设计，便于后续做 MQ 幂等（当前实现以 REST 为主）。

2. 为什么这样设计，如何设计实现的（关键细节）

2.1 整体架构思路

  • 注解驱动：用 @Idempotent 标记需要幂等的方法，业务无侵入。
  • 场景 + 类型二维选择：IdempotentScene（如 RESTAPI/MQ）× IdempotentType（TOKEN/PARAM/SPEL），不同组合对应不同策略。
  • 策略可插拔：每种策略一个 Handler，由工厂按注解参数选择，便于扩展新场景（如 MQ）或新类型。

这样设计的原因：幂等“如何识别同一请求”在不同场景下差异大（Token 一次性、参数 MD5、业务主键等），用“场景 + 类型”可以清晰区分，并用独立 Handler 实现，避免在一个类里堆满 if-else。

2.2 核心流程（AOP + 模板方法）

请求进入 → @Idempotent 切面 → 根据 scene/type 取 Handler → execute(joinPoint, idempotent)
         → buildWrapper() 构建幂等键/锁键
         → handler() 执行幂等校验（取锁 / 校验 Token / 等）
         → 校验通过则 proceed() 执行业务
         → 若抛 RepeatConsumptionException 且 error==false 可返回 null（为 MQ 预留）
         → 若业务异常则 exceptionProcessing()（如释放锁）
         → 最后 postProcessing()（释放锁） + IdempotentContext.clean()

要点：

  • 切面顺序：@Order(Ordered.HIGHEST_PRECEDENCE)，保证幂等校验优先于其他切面（如日志、权限）。
  • 模板方法：AbstractIdempotentExecuteHandler 中 execute() 先 buildWrapper() 再 handler()，子类只实现“如何建键”和“如何校验”，锁的释放放在 postProcessing()/exceptionProcessing()，避免重复写 finally。

2.3 三种幂等类型的关键实现

2.3.1 TOKEN（一次性令牌）

  • 思路：先调用接口拿 Token，再在业务请求头里带 Idempotent-Token；服务端校验到 Token 存在则删除并放行，删除失败或缺失则认为是重复/非法。
  • 关键细节：
    • Token 生成：prefix + UUID，存入 Redis，key 为 idempotent:token: + token，带过期时间（配置或默认 60s）。
    • 校验时用 DistributedCache.delete(TOKEN_PREFIX_KEY + token)，只有删除成功才放行，保证一 Token 只能用一次。
    • 未带 Token 或 Token 已用/过期会抛 ClientException，提示信息可用 @Idempotent.message() 配置。

2.3.2 PARAM（基于请求参数）

  • 思路：用“服务名 + 请求路径 + 请求体 MD5”生成唯一键，对该键抢分布式锁；抢到锁则视为首次请求并执行业务，抢不到则视为重复。
  • 关键细节：
    • Key 格式：idempotent:service:[applicationName]:path:[servletPath]:md5:[argsMD5]，保证同接口、同参数会命中同一把锁。
    • 使用 Redisson RLock.tryLock()，不阻塞；拿不到锁直接抛 ClientException(message)。
    • 锁在 IdempotentContext（ThreadLocal）中保存，在 postProcessing() 中统一 unlock()，避免业务异常导致锁泄漏。

2.3.3 SPEL（基于 SpEL 表达式）

  • 思路：由业务在注解里写 SpEL（如 #id、#request.orderId），从方法参数中解析出业务唯一键，再对该键加锁。
  • 关键细节：
    • buildWrapper 里通过 SpELUtil.parseKey(idempotent.key(), method, args) 得到 lockKey，再可加 uniqueKeyPrefix、keyTimeout（为 MQ 等场景预留）。
    • 与 PARAM 类似，用 Redisson 对 uniqueKeyPrefix + lockKey 加锁，成功则放入 IdempotentContext，在 postProcessing() 释放；并在 exceptionProcessing() 里也释放锁，避免业务异常时锁不释放。

2.4 锁与上下文的统一管理

  • IdempotentContext：ThreadLocal<Map<String, RLock>>，存当前请求在本线程内拿到的幂等锁（如 PARAM 的 idempotent:param:restApi、SPEL 的 lock:spEL:restAPI），便于在 postProcessing()/exceptionProcessing() 中统一释放。
  • finally 中 clean()：每个请求结束（成功或异常）都会执行 IdempotentContext.clean()，避免 ThreadLocal 泄漏。

2.5 工厂与自动装配

  • IdempotentExecuteHandlerFactory：根据 IdempotentScene + IdempotentType 从 Spring 容器取对应 Bean（如 RESTAPI + PARAM → IdempotentParamService），当前仅实现 RESTAPI 下的三种 type。
  • IdempotentConfiguration：通过 spring.factories 自动装配，注册切面、三种 Handler（以 Service 接口暴露）、Token 控制器；并 @EnableConfigurationProperties(IdempotentProperties.class) 绑定 framework.idempotent.token.prefix/timeout。

2.6 为 MQ 预留的设计

  • IdempotentScene.MQ：枚举已定义，工厂中未实现具体 Handler，便于后续加 MQ 专用 Handler。
  • RepeatConsumptionException(error)：Aspect 中若捕获到且 !ex.getError() 可返回 null，注释说明用于“消息已处理成功，直接返回成功”的场景，与“消息处理中需重试”区分，方便对接 RocketMQ 等重试语义。
  • uniqueKeyPrefix / keyTimeout：在 SPEL 上预留，MQ 场景可按消息 ID 或业务键 + 前缀/过期时间做幂等。

3. 这样设计的优点

维度 说明
业务无侵入 仅加 @Idempotent 注解即可，键的生成、锁的获取与释放、异常处理都在框架内完成。
策略清晰可扩展 场景 × 类型对应不同 Handler，新增 MQ 或新类型只需新增 Handler 并在工厂中挂接，符合开闭原则。
统一抽象 IdempotentExecuteHandler 的 execute/handler/postProcessing/exceptionProcessing 把“建键 → 校验 → 执行业务 → 收尾”固定下来，避免各实现类逻辑分散。
锁安全 锁放入 ThreadLocal，在 finally 中 postProcessing 释放，异常时 exceptionProcessing 也释放，降低锁未释放风险。
可配置 Token 前缀、过期时间、SpEL key、message 等均可配置或注解指定，适配不同业务。
与现有技术栈一致 依赖 Redisson、分布式缓存、Spring 生态，和项目内其他模块风格统一。

4. 使用方法介绍

4.1 依赖与配置

  • 引入模块依赖（如 framework-biz-idempotent），该模块已依赖 framework-web、framework-cache。
  • 自动装配：引入后无需额外开关，IdempotentConfiguration 会通过 spring.factories 加载。
  • 可选配置（如 application.yml）：
framework:
  idempotent:
    token:
      prefix: "idempotent:"   # Token 前缀，可选
      timeout: 60000         # Token 过期时间（毫秒），可选

4.2 Token 方式（REST API，一次性令牌）

  1. 先调接口获取 Token（框架提供）：
    • 路径：GET {框架路径}/idempotent/createToken（如 ServletConstants.DEFAULT_FRAMEWORK_PATH 为 /api 时，为 GET /api/idempotent/createToken）。
  2. 在业务请求头中携带该 Token：
    • Header：Idempotent-Token: <token>。
  3. 在需要防重的接口上使用注解：
@PostMapping("/create")
@Idempotent(type = IdempotentType.TOKEN, message = "操作过于频繁，请稍后再试")
public Result<OrderDTO> createOrder(@RequestBody OrderRequest request) {
    // 订单创建逻辑
    return Result.success(orderDTO);
}

4.3 PARAM 方式（REST API，同接口同参数防重）

根据“服务名 + 路径 + 请求体 MD5”自动生成幂等键，无需前端传 Token：

@PostMapping("/pay")
@Idempotent(type = IdempotentType.PARAM, message = "支付请求正在处理中，请稍后再试")
public Result<String> pay(@RequestBody PaymentRequest request) {
    // 支付逻辑
    return Result.success("支付成功");
}

4.4 SPEL 方式（REST API，按业务键防重）

从方法参数中通过 SpEL 提取唯一键，适合“同一业务主键只处理一次”的场景：

@PostMapping("/update/{id}")
@Idempotent(type = IdempotentType.SPEL,
            key = "#id + ':' + #request.version",
            message = "用户信息更新请求正在处理中，请稍后再试")
public Result<UserDTO> updateUser(@PathVariable Long id, @RequestBody UserUpdateRequest request) {
    // 用户更新逻辑
    return Result.success(userDTO);
}

注解属性简要说明：

  • key：SpEL 表达式，仅 type=SPEL 时生效。
  • message：幂等校验失败时返回的提示。
  • type：TOKEN / PARAM / SPEL。
  • scene：当前使用默认 RESTAPI 即可。
  • uniqueKeyPrefix / keyTimeout：可选，多用于 MQ 或需要自定义 Redis key 前缀与过期时间时。

4.5 异常处理建议

幂等校验失败会抛出 ClientException（或带错误码的异常），建议在全局异常处理中统一封装返回：

@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ClientException.class)
    public Result<String> handleClientException(ClientException e) {
        return Result.failure(e.getErrorCode(), e.getMessage());
    }
}

5. 归纳总结

  • 解决的问题：在 REST API（及预留 MQ）场景下，防止同一操作被重复执行导致的数据与资金不一致；支持“一次性 Token”“同参数防重”“按业务键防重”三种方式。
  • 设计要点：注解驱动 + 场景/类型二维选择 + 策略模式（Handler 工厂）+ 模板方法（建键 → 校验 → 执行业务 → 收尾）；锁与上下文统一管理，异常与正常出口都释放锁并清理 ThreadLocal。
  • 实现要点：TOKEN 用 Redis 存删实现一次一用；PARAM 用“服务+路径+参数 MD5”+ Redisson 锁；SPEL 用表达式解析业务键再加锁；通过 IdempotentContext 和 postProcessing/exceptionProcessing 保证锁不泄漏。
  • 使用方式：引入依赖即可生效；REST 推荐 TOKEN 或 PARAM，需要按业务主键时用 SPEL；配置 Token 前缀与超时可选；对 ClientException 做全局处理即可。

该模块在保证接口幂等、防止重复提交与重复消费方面可直接用于生产，后续扩展 MQ 场景时只需增加 MQ 的 Handler 并在工厂中按 IdempotentScene.MQ 分支即可。
